# Research: Ceipal ATS v1 API — auth + recruitment endpoints

> Compiled 2026-06-30 via WebSearch (the developer portal returned HTTP 403 to
> direct fetch, so specifics below come from the search index + domain knowledge;
> re-run via research.py once Ollama is fixed).

## Where the docs live
- Official developer portal: **developer.ceipal.com**, with an **ATS v1 API
  Reference**, an interactive **API Explorer / API Runner**, and a changelog [1][2][3].

## Endpoints (ATS v1)
The v1 reference documents these resources [2]:
- **Authentication** and **Refresh Token**
- **Job Postings** and **Job Posting Details**
- **Applicants** and **Applicant Details** (candidate search / profile)
- **Submissions** and **Submission Details**
- **Talent Bench**, **Clients**, **Leads**, **Vendors**, **Interviews**,
  **Placements**, **Users**, **Master Data**

## Authentication model
- Token-based: authenticate (Ceipal uses an **email + password + API key** to mint
  an **access token**), then use a **Refresh Token** endpoint to renew it [2]. Send
  the token as a bearer header on subsequent calls.

## Cost / access reality
- **Ceipal has no public free tier** and the portal is gated (our fetch got 403).
  So for this $0 project, **Ceipal stays mocked**; the real adapter is a thin,
  env-gated stub we can finish if the owner supplies sandbox credentials.

## Implications for this project
- `AtsPort` (Ceipal) methods: `search_candidates`, `get_candidate`, `list_jobs`,
  `get_job`, `create_submission`, `update_submission_stage`, `save_screening_outcome`
  (writes the screening result + RTR back).
- **MockCeipal** implements these over SQLite (`Candidate`, `Job`, `Submission`,
  `RTR`). The real **CeipalAdapter**: `POST` auth (email/password/api_key) → token →
  call **Applicants** (search), **JobPostings** (reqs), **Submissions** (submit).
  Field names map 1:1 to our domain models.
